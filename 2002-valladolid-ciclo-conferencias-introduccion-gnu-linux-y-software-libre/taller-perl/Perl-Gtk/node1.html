1. ¿Qué es GTK?


GTK es el GIMP Toolkit, un conjunto de herramientas para crear interfaces gráficas. GTK es multiplataforma, y ofrece toda una jerarquía de widgets para el desarrollo de aplicaciones gráficas.

Es GNU, y su licencia LGPL por lo que además de poder usarse para el desarrollo de software propietario lo podemos usar en nuestras aplicaciones open source.

Inicialmente se desarrollo para el GIMP (GNU Image Manipulation Program) que es un programa de tratamiento de imagenes muy popular en la plataforma Linux.

Está muy asociado al entorno GNOME.

Aunque inicialmente se encontraba disponible para C/C++ en la actualidad muchos lenguajes de programación tienen módulos o extensiones para programar en GTK. Entre los lenguajes en los que podemos usar las librerías GTK están C, C++, Perl, Python, Eiffel, ...

En Perl se usa a través de un módulo llamado Gtk. La mayoría de las distribuciones lo incluyen en algún paquete con un nombre como Gtk-Perl.rpm o algún otro nombre parecido.
 
 
 

2. Antes de empezar


Lo primero de todo es asegurarnos de que el módulo Gtk está instalado. En la mayoria de las distribuciones Linux suele venir como un paquete (rpm o deb) con el nombre Gtk-Perl o alguno similar.

Si no, siempre podemos ir a CPAN (www.cpan.org) o por supuesto de la misma página de GTK (www.gtk.org) para obtenerlo. Para detalles de cómo instalar el módulo cuando no viene en un paquete de distribución (rpm o deb) ver la sección de módulos de cpan del tutorial de Perl Básico
 
 
 

3. Unos conceptos básicos


Antes  de meternos de lleno con la programación GTK, vamos a dar unos conceptos básicos sobre la programación de interfaces de usuario:

Casi todos los lenguajes de desarrollo de interfaces graficas de usuario (GUI) modernos están orientados a objetos. La razón para ello es que todos los conceptos que aparecen en la teoría de diseño de interfaces como botones, ventanas, barras de scroll, etc... se prestan muy bien a ser representados por objetos de un lenguaje de programación y poder así aprovechar propiedades como el encapsulamiento de atributos o la herencia.

A los objetos visuales se les suele denominar  widgets, aunque no es una norma general ni estandarizada. Un widget sería así un boton, una barra de herramientas, un scrollbar, etc... Como estamos en orientación a objetos, habrá con toda seguridad una clase Widget o con otro nombre  de la cual desciendan todas las demás. La mayor parte de las librerías gráficas también proveen de objetos no visuales como threads o listas enlazadas. Entonces es bastante problable que esos objetos no desciendan de la clase widget.

La interfaz se puede ver como un conjunto de objetos que responden a señales o eventos. Un evento se produce por ejemplo cuando el usuario realiza una determinada acción sobre la interfaz (por ejemplo: hacer click en un botón, o mover el ratón por la ventana).

Una vez que un evento se produce podemos querer que se realize alguna acción o no como respuesta a ese evento. En caso afirmativo tendremos que programar un manejador de evento, que normalmente quiere decir que tenemos que hacer una subrutina o función que contenga el código que queremos que se ejecute cuando ocurra el evento.

Lo siguiente que queda es escoger el lenguaje de programación que sea y aprenderse su jerarquia de widgets (es decir conocer que tipo de objetos gráficos tenemos y las relaciones entre ellos).
